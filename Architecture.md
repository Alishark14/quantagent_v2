**PART I --- FOUNDATIONS**

**1. Executive Summary**

QuantAgent v2.0 is a complete architectural redesign of the QuantAgent
trading engine, evolving from a 3-layer LLM pipeline into a 5-layer
hybrid intelligence platform capable of trading any financial instrument
across any exchange that provides an API.

The core innovation is separating signal generation from signal
evaluation from trade execution, introducing a ConvictionAgent that
scores signal quality on a continuous 0--1 scale before any trade
decision is made. This replaces the current binary 2/3 voting system
with a nuanced, regime-aware evaluation that accounts for market
structure, positioning data, multi-timeframe context, and learned
lessons from past trades.

**1.1 Vision**

Trade ALL financial markets (cryptocurrencies, stocks, ETFs,
commodities, forex, options, perpetual futures, and traditional futures)
across multiple exchanges via APIs. The engine operates as a universal
trading intelligence layer that adapts to any market and any asset
class.

**1.2 Business Goals**

**Goal 1 --- Consumer Product:** Make AI-powered trading bots accessible
to everyone. Users connect their exchange, configure a strategy, and the
engine handles analysis, execution, and risk management through a
managed dashboard and iOS app.

**Goal 2 --- B2B Engine API:** Sell the engine as a stateless API to
quantitative traders, financial institutions, and individual investors.
Clients send market data, receive conviction-scored signals and trade
recommendations. They manage their own execution.

**Goal 3 --- Data & Intelligence Products:** Monetize the data moat.
Sell labeled datasets (chart images, decision traces, MFE/MAE
distributions), alpha factor subscriptions, regime-labeled market data,
and fine-tuned model inference. The data generated by Goals 1 and 2
becomes a standalone product line.

**2. Design Principles & Modular Architecture**

Every major subsystem communicates through defined contracts
(interfaces), never through direct internal access. Modules publish and
subscribe to typed events via an Event Bus. Any module can be swapped,
upgraded, disabled, or replaced without affecting others, as long as the
replacement emits the same event types with the same payload schemas.

**2.1 Event Bus**

The Event Bus is the central nervous system. Modules never call each
other directly. The DataModule emits DATA_READY with a MarketData
payload; the SignalModule subscribes to DATA_READY and runs when data
arrives. If SignalModule is replaced entirely, nothing else changes as
long as it emits SIGNALS_READY with the correct schema.

For single-server deployment, the Event Bus is an in-process
publish/subscribe system (Python asyncio queues). For multi-server
deployment, it uses Redis Pub/Sub or RabbitMQ. Same interface, different
backend --- configured at startup, invisible to modules.

**2.3 Plugin Architecture Within Modules**

Each module is itself modular via plugin registries. The SignalModule
does not hardcode three agents --- it has a SignalProducer registry.
Adding a 4th agent, removing one, or replacing an LLM agent with an ML
model is a configuration change, not a code change.

> class SignalModule:
>
> def \_\_init\_\_(self, producers: list\[SignalProducer\]):
>
> self.producers = producers \# injected, not hardcoded
>
> async def analyze(self, data: MarketData) -\> list\[SignalOutput\]:
>
> tasks = \[p.analyze(data) for p in self.producers
>
> if p.is_enabled()\]
>
> results = await asyncio.gather(\*tasks)
>
> return \[r for r in results if r is not None\]

The same pattern applies to: ExchangeAdapter (exchange plugins),
FlowProvider (flow data plugins), ExternalSignalProvider (API plugins),
NotificationChannel (distribution plugins), and StorageBackend (database
plugins).

**2.4 Feature Flags**

Every module and sub-component can be enabled or disabled via feature
flags without code changes. Flags are stored in configuration
(environment variables for simple deployments, database for
multi-tenant). The system gracefully handles disabled modules ---
downstream modules receive empty inputs and adjust behavior accordingly.

**2.1 Design Principles**

1.  **Engine as a library.** The analysis pipeline (Data → Signal →
    Conviction → Decision) is a pure function: data in, signal out. No
    web framework, no database, no scheduler dependencies in the core.
    Consumer platform and B2B API are wrappers around the same engine.

2.  **Fact vs. Subjective separation.** Every input to ConvictionAgent
    is explicitly labeled as factual (computed, deterministic) or
    subjective (LLM-interpreted, can hallucinate). Regime determines
    which category gets more weight.

3.  **Hybrid signal architecture.** The Signal Layer accepts both LLM
    agents and ML models as signal producers via a unified
    SignalProducer interface. ML models add zero latency (milliseconds
    vs. seconds for LLM) and zero marginal cost.

4.  **Exchange is the only source of truth.** Position sync queries
    exchange periodically and fixes any wrong statuses in the database.
    Internal records never override exchange state.

5.  **SKIP is always safe.** On any error, ambiguity, parse failure, or
    low conviction in the trading pipeline, the system defaults to SKIP.
    Never defaults to LONG or SHORT.

6.  **Design for universal, implement for one.** Architecture supports
    any market and exchange from day one. Capabilities system allows
    graceful degradation when features (native SL/TP, GEX data, market
    hours) are unavailable.

7.  **Parameters are regime-driven, not LLM-driven.** The LLM identifies
    market regime (which it is good at). Code translates regime into
    optimal ATR multipliers, RR ratios, and risk parameters (which math
    is good at).

8.  **External APIs are enrichment, not dependencies.** Every external
    signal provider is optional. If any goes down, the engine still
    works with local computation. External signals improve quality but
    never gate operation.

**PART II --- ENGINE**

**4. Data Layer**

The Data Layer is entirely code-based with zero LLM calls. It gathers,
computes, and structures all data that downstream layers consume. Every
output is deterministic and reproducible.

**4.1 OHLCV Fetcher**

Fetches candlestick data from the appropriate source based on asset
type. The candle count is timeframe-dependent to ensure each chart
captures meaningful market structure:

  -------------------------------------------------------------------------------
  **Timeframe**   **Candles**   **Coverage**    **Rationale**
  --------------- ------------- --------------- ---------------------------------
  15m             100           \~25 hours      One day of context, sufficient
                                                for scalp setups

  30m             100           \~2 days        Good for intraday patterns

  1h              150           \~6 days        Captures weekly structure

  4h              150           \~25 days       Full monthly cycle visible

  1d              200           \~10 months     Major trend structure
  -------------------------------------------------------------------------------

Data sources are selected per asset type via a DataProvider registry:
Bybit or Binance for crypto OHLCV, Hyperliquid for HIP-3 markets,
Polygon.io for US equities, and the exchange itself for any asset
without a dedicated data provider.

**4.2 Parent Timeframe Context**

Every bot fetches a second set of candles from the parent timeframe (50
candles) and computes trend context locally. A 1h bot checks the 4h
trend; a 4h bot checks the 1d trend. This provides macro directional
bias without any LLM call.

Computed locally: MA direction (price above/below 50-period MA), ADX
value and classification (trending \>25 / ranging \<20), Bollinger Band
width percentile (volatility context). Output is a structured object
injected into both ConvictionAgent and the signal agents as grounding
context.

  -----------------------------------------------------------------------
  **Bot TF**        **Parent TF**     **Parent          **Parent
                                      Candles**         Coverage**
  ----------------- ----------------- ----------------- -----------------
  15m               1h                50                \~2 days

  30m               4h                50                \~8 days

  1h                4h                50                \~8 days

  4h                1d                50                \~2.5 months

  1d                1w                50                \~1 year
  -----------------------------------------------------------------------

**4.3 Indicator Calculator**

Computes all technical indicators locally from OHLCV data. These are
factual, deterministic values that serve as ground truth for grounding
vision agents and as direct input to IndicatorAgent. Indicators
computed: RSI (14), MACD (12/26/9), ROC (10), Stochastic Oscillator
(14/3/3), Williams %R (14), ATR (14), ADX (14), Bollinger Bands (20/2),
and Volume MA (20).

**4.4 Swing Detection**

The existing swing_detection.py module identifies significant swing
highs and lows from the last 50 candles. These levels serve multiple
purposes: structural stop-loss placement, pyramid distance-to-resistance
checks, and grounding context for signal agents. Output includes the 3
nearest swing highs and 3 nearest swing lows with their price levels.

**4.5 FlowAgent (Code, No LLM)**

FlowAgent computes market positioning and flow data as structured
output. It is NOT an LLM agent. It operates through pluggable data
providers that activate based on asset type and data availability.

**5. Signal Layer**

The Signal Layer produces directional assessments of market conditions.
All signal producers (LLM agents and ML models) run in parallel and
produce outputs conforming to a unified SignalOutput interface. ML
models execute in milliseconds while LLM calls take 1--3 seconds, so ML
models add zero latency to the cycle.

**5.1 SignalProducer Interface**

Every signal producer, whether LLM or ML, implements the same abstract
interface. ConvictionAgent receives a list of SignalOutput objects and
does not care whether they came from Claude or XGBoost.

> class SignalProducer(ABC):
>
> def name(self) -\> str: \...
>
> def signal_type(self) -\> str: \# \'llm\' \| \'ml\'
>
> async def analyze(self, data: MarketData) -\> SignalOutput: \...
>
> \@dataclass
>
> class SignalOutput:
>
> direction: str \# BULLISH \| BEARISH \| NEUTRAL \| None
>
> confidence: float \# 0.0 to 1.0
>
> reasoning: str \# text explanation (LLM) or features (ML)
>
> signal_category: str \# \'directional\' \| \'regime\' \| \'anomaly\'
>
> data_richness: str \# \'full\' \| \'partial\' \| \'minimal\'
>
> contradictions: str \# noted conflicts with grounding data
>
> key_levels: dict \# support/resistance identified

**5.2 LLM Signal Agents**

Three Claude Sonnet agents run in parallel. Each receives a grounding
context header containing factual data (indicator values, flow data,
parent TF bias) that anchors their analysis in mathematical reality.
This solves the \"vision majority vulnerability\" where two vision
agents could outvote hard math.

**5.2.1 Grounding Context Header**

Injected into every LLM signal agent before their specialized input:

> CONTEXT (do not override with visual impression):
>
> Symbol: BTC-USDC \| Timeframe: 1h \| Lookback: 150 candles (\~6 days)
>
> Forecast horizon: 3 candles (\~3 hours)
>
> RSI: 73.2 (overbought) \| MACD: histogram declining, bearish cross
>
> Stochastic: 82 (overbought) \| Williams %R: -18 (overbought)
>
> Funding rate: +0.042% (crowded long) \| OI change: +8.2% (4h)
>
> Parent TF (4h): BEARISH, price below 50-MA, ADX 31 (strong trend)
>
> Nearest resistance: \$65,400 (0.8 ATR away)
>
> Nearest support: \$63,100 (2.1 ATR away)

**5.2.2 IndicatorAgent (Text, \~\$0.008)**

Receives computed indicator values as text. Produces momentum
assessment, overbought/oversold classification, divergence detection,
and crossover signals. System prompt explicitly names all five
indicators, specifies the lookback window and forecast horizon, and asks
the agent to note contradictions with the grounding context.

**5.2.3 PatternAgent (Vision, \~\$0.009)**

Receives a candlestick chart image. The 16-pattern library remains in
the system prompt (cacheable). The agent reports detected patterns,
completion percentage, directional bias, and must reconcile visual
impressions with the grounding context. Output includes a
contradictions_noted field when the visual pattern conflicts with
indicator readings.

**5.2.4 TrendAgent (Vision, \~\$0.010)**

Receives a chart with OLS trendlines overlaid. Analyzes price-trendline
interactions (bounce, break, compress), channel characteristics
(widening, narrowing, parallel), and trend regime assessment (clean
trend, choppy range, volatile expansion). The visual regime assessment
complements the computed regime from parent timeframe data.

ML models are implemented as SignalProducers that return SignalOutput
objects. They are designed as slots: the architecture supports them from
day one, but they return null until trained. ConvictionAgent ignores
null-returning producers.

**6. Conviction Layer**

The Conviction Layer is the critical architectural innovation of v2.0.
It replaces the current binary 2/3 voting system with a nuanced
meta-evaluation that produces a continuous conviction score (0--1), a
regime classification, contradiction analysis, and a reasoning trace.

ConvictionAgent is the only component that sees the complete picture:
all signal agent reports, all flow data, parent timeframe context, raw
indicator values, swing levels, and learned rules from past trades. Its
job is NOT to generate a new signal, but to evaluate the quality,
coherence, and reliability of existing signals.

ConvictionAgent receives explicitly labeled factual vs. subjective
inputs:

> ATR)

**6.3 Regime-Adaptive Weighting**

ConvictionAgent adjusts its trust in factual vs. subjective inputs based
on the detected regime. This implements the FS-ReasoningAgent research
finding that factual signals outperform in bear/ranging markets while
subjective pattern recognition outperforms in trending markets.

  ---------------------------------------------------------------------------
  **Regime**        **Factual     **Subjective    **Rationale**
                    Weight**      Weight**        
  ----------------- ------------- --------------- ---------------------------
  TRENDING_UP       40%           60%             Patterns and trends are
                                                  reliable in strong trends

  TRENDING_DOWN     50%           50%             Balanced; trend exhaustion
                                                  signals matter

  RANGING           70%           30%             Indicators (RSI, Stoch)
                                                  excel at mean-reversion

  HIGH_VOLATILITY   60%           40%             Trust math over visual
                                                  noise in chaos

  BREAKOUT          30%           70%             Pattern completion drives
                                                  breakout trades
  ---------------------------------------------------------------------------

**6.4 Conviction Thresholds**

The conviction score directly drives DecisionAgent behavior:

  -----------------------------------------------------------------------------
  **Score       **Classification**   **DecisionAgent Behavior**
  Range**                            
  ------------- -------------------- ------------------------------------------
  0.0 -- 0.3    LOW                  SKIP always. No trade regardless of
                                     direction.

  0.3 -- 0.5    MARGINAL             SKIP unless 3/3 LLM agent agreement AND
                                     favorable flow.

  0.5 -- 0.7    MODERATE             Trade allowed. Base position size. No
                                     pyramiding.

  0.7 -- 0.85   HIGH                 Trade allowed. Full position size.
                                     Pyramiding permitted.

  0.85 -- 1.0   VERY HIGH            Trade allowed. Enhanced size (1.3x).

**7.2.3 Volatility Percentile Scaling**

On top of regime adjustment, ATR multiplier scales with current
volatility percentile (comparing current ATR to the last 100 ATR
values). Above 80th percentile: multiply by 1.15 (extra wide). Below
20th percentile: multiply by 0.85 (tighter). This prevents stops from
being too tight in high-volatility environments or too wide in calm
markets.

**7.3 Mechanical Safety Checks (Code, Not LLM)**

These checks execute after DecisionAgent outputs its action but before
any order is placed. They are deterministic, non-negotiable, and cannot
be overridden by the LLM.

1.  **Pyramid distance-to-resistance gate:** If ADD_LONG and price is
    within 0.3×ATR of nearest swing high, convert to HOLD. Prevents
    adding at the local top.

2.  **Daily loss limit:** If realized + unrealized P&L for the day
    exceeds configured max loss, block all new entries. HOLD and
    CLOSE_ALL still permitted.

3.  **Position count limit:** One position per symbol per bot. Checked
    before every trade.

4.  **SL placement failure:** If stop-loss order placement fails,
    immediately close the position. No unprotected positions.

5.  **Conviction floor:** If conviction score \< 0.3, force SKIP
    regardless of DecisionAgent output. Hard-coded safety.

**7.4 Position Sizing**

Position size is now conviction-scaled on top of existing volatility and
agreement logic:

> base = budget / num_symbols / max_concurrent_positions
>
> vol_adjusted = base \* clamp(avg_atr / current_atr, 0.5, 1.5)
>
> conviction_scaled = vol_adjusted \* conviction_multiplier(score)
>
> // score 0.5-0.7: x1.0, 0.7-0.85: x1.15, 0.85-1.0: x1.3
>
> final = clamp(conviction_scaled, \$20, budget \* max_position_pct)

**7.5 Execution Cost Model (Zero-Leakage Execution)**

The ExecutionCostModel computes the total execution tax on every trade
BEFORE the trade is placed. If the math doesn't work, the trade is
physically blocked by Safety Check #6. This prevents bots from grinding
to zero on execution costs --- especially on HIP-3 assets where
deployer fees, wide spreads, and thin orderbooks can eat 20--50% of a
small position's risk budget.

Total Execution Cost = Taker Fee (Entry) + Estimated Slippage
(orderbook-based) + Estimated Spread (bid-ask) + Estimated Funding
(hold-time projected) + Taker Fee (Exit).

**7.5.1 Exchange-Agnostic Design**

ExecutionCostModel is an abstract interface. Each exchange provides its
own implementation. When no exchange-specific model exists, a
GenericCostModel with conservative defaults (0.1% taker, 0.05%
slippage, 0.1% spread) ensures safety.

  ---------------------------------------------------------------------------
  **Exchange**       **Cost Model**             **Key Modifiers**
  ------------------ -------------------------- -----------------------------
  Hyperliquid        HyperliquidCostModel       HIP-3 deployer fees (0--300%),
                                                growth mode (90% reduction),
                                                staking tiers (5--40% off),
                                                referral discounts

  Binance            BinanceCostModel           BNB discount, VIP tiers

  IBKR               IBKRCostModel              Commission-based, no funding

  Generic            GenericCostModel           Conservative defaults
  ---------------------------------------------------------------------------

**7.5.2 Dynamic Asset Metadata (No Hardcoding)**

Asset fee profiles are NEVER hardcoded. The HyperliquidCostModel fetches
all asset metadata (deployer fee scales, growth mode, HIP-3 status) from
the Hyperliquid meta API endpoint and caches it for 24 hours. When
Hyperliquid lists a new asset or changes a deployer fee, the engine picks
it up automatically. Zero maintenance.

**7.5.3 Cost Components**

ExecutionCost captures: entry taker fee, exit taker fee, entry slippage
(estimated from orderbook depth walk), exit slippage, bid-ask spread
cost, estimated funding over expected hold duration, total cost (sum),
and total as percentage of position value.

**7.5.4 Circular Position Sizing (Iterative Solver)**

Standard risk formulas compute position size as Risk / SL Distance. But
execution costs are a percentage of position size, creating circular
math. The ExecutionCostModel uses an iterative solver that converges in
2--3 iterations: starting with a fee-blind estimate, computing costs at
that size, then reducing the size so (trade loss + ALL costs) = maximum
permitted risk. If costs exceed the entire risk budget, position size
returns zero and the trade is blocked.

**7.5.5 Fee-Adjusted R:R**

Every trade computes both raw R:R (pure price movement) and fee-adjusted
R:R (net of all execution costs). The grounding header sent to
ConvictionAgent includes a single distilled line:

> COST_AWARE_RR: SL 1.5% / TP 3.0% -> raw RR 2.00 -> net RR 1.52. Fee
> drag: HIGH (24%).

Three severity levels: LOW (fee drag < 5%), MEDIUM (5--15%), HIGH (>
15%). The LLM sees the mathematical reality without needing to understand
why a fee is high.

**7.5.6 Safety Check #6: Cost Viability Gate**

A mechanical, non-negotiable safety check runs after DecisionAgent but
before order placement. If the fee-adjusted R:R falls below the
timeframe's minimum R:R, the trade is converted to SKIP. No LLM can
override this. This is the "Zero-Leakage Execution" differentiator ---
the engine physically cannot enter unprofitable trades.

  -----------------------------------------------------------------------
  **Scenario**                **Fee Drag**   **Action**
  --------------------------- -------------- ----------------------------
  BTC $1000, 1h               3% (LOW)       Trade proceeds normally

  GOLD HIP-3 $200, 1h         24% (HIGH)     Blocked by Check #6

  BTC $50, 15m                12% (MEDIUM)   Proceeds if R:R still meets
                                              minimum after costs

  Any asset, costs > risk     100%           Position size = 0, blocked
  budget
  -----------------------------------------------------------------------

**7.5.7 Dual Execution Mode**

For entries triggered by the Sentinel, pure taker (market) orders are
used because momentum is happening and speed matters. For post-trade
SL/TP management, the Sentinel Position Manager can replace trigger
orders with maker limit orders on the book, earning maker rebates
instead of paying taker fees on exit. At Tier 0 this saves 0.06% per
exit. For B2B API customers, both taker and maker cost estimates are
returned, and clients choose their own execution routing.

**7.5.8 Expected Hold Duration**

Funding cost estimation requires expected hold time. This varies by
timeframe:

  -----------------------------------------------------------------------
  **Timeframe**   **Expected Hold**   **Funding Periods (8h)**
  --------------- ------------------- -------------------------
  15m             2 hours             0.25
  30m             4 hours             0.5
  1h              8 hours             1.0
  4h              24 hours            3.0
  1d              72 hours            9.0
  -----------------------------------------------------------------------

For 4h and 1d bots, funding cost can be substantial. A 0.03% funding
rate over 72 hours = 0.27% position cost from funding alone. The cost
model captures this and factors it into the viability check.

**8.4 Scheduled Cycle Fallback**

The scheduled cycle still exists as a guaranteed minimum. Even if no
Sentinel event fires, the bot runs its full analysis at least once per
configured interval (e.g., every 1h). This ensures: open positions are
monitored for HOLD/CLOSE_ALL decisions, memory is updated, and the
system never goes silent. The scheduled cycle and Sentinel-triggered
cycles run through the same pipeline --- only the trigger mechanism
differs.

**8.5 Sentinel for B2B API**

**9.3 Orphan Position Reaper**

If a TraderBot crashes mid-trade (after placing an order but before
SL/TP is confirmed), an orphan position exists on the exchange without
active management. The Reaper is a background process (running in the
dashboard backend, similar to existing Position Sync) that periodically
checks for positions on the exchange that have no living TraderBot
managing them.

When an orphan is detected: if SL/TP orders exist on the exchange, the
position is safe (exchange manages it). If no SL/TP exists, the Reaper
immediately places a wide emergency SL (2x ATR) and alerts the operator.
It then optionally spawns a new TraderBot to take over management of the
orphaned position.

**9.4 Consumer Product UX**

For consumer users, the ephemeral model changes the UX metaphor. Users
do not \"start a bot.\" They \"watch a market.\" The Sentinel watches on
their behalf. When it sees opportunity, it spawns a trader, takes the
trade, and reports back. The user sees a clean activity feed:
\"BTC-USDC: 3 trades today, +\$47 net. Current: watching for next
setup.\" This is simpler, more intuitive, and feels more intelligent
than \"Bot is running\... Bot is sleeping\... Bot is running\...\"

**9.5 Cost Advantage**

Instead of paying \~\$1.08/day for a 1h bot running 24 cycles regardless
of market conditions, costs scale with opportunity. In a quiet market:
5--8 full analyses (\$0.22--\$0.36). In an active market: 15--20
analyses (\$0.67--\$0.90). The system naturally spends more on analysis
when there are more opportunities and less when the market is quiet.
This is a significant cost efficiency improvement.

**10. Memory System**

The v2.0 memory system operates through four distinct feedback loops,
each serving a different temporal scope and learning objective. Memory
is what makes the system intelligent over time rather than stateless per
cycle.

**10.1 Loop 1: Cycle Memory (Per-Bot, Short-Term)**

Carried over from v1.1. Stored in the database per bot, survives
restarts.

**Contents:** Last 5 cycle decisions with agent signals, current
position details (direction, avg entry, size, pyramid count, age), last
closed trade result. Injected into DecisionAgent every cycle for
position management decisions (pyramid, hold, close).

**Also injected into:** ConvictionAgent, for pattern detection across
recent cycles (e.g., declining conviction scores over last 3 cycles =
fading momentum).

**10.2 Loop 2: Reflection Memory (Per-Asset, Medium-Term)**

Generated asynchronously by ReflectionAgent after each trade closes.
Stored as a JSON rulebook per asset and timeframe.

**10.2.1 ReflectionAgent Process**

When a trade closes (win or loss), ReflectionAgent (1 LLM call,
\~\$0.010) receives the original entry reasoning, the agent signals at
entry, the flow data at entry, and the actual trade outcome. It compares
expectation vs. reality and distills a one-line rule if a pattern is
identified.

**10.2.2 Rule Structure**

> {
>
> \"rules\": \[
>
> {
>
> \"id\": \"rule_042\",
>
> \"created\": \"2026-04-05\",
>
> \"asset\": \"ETH-USDC\",
>
> \"timeframe\": \"1h\",
>
> \"rule\": \"Avoid LONG when RSI \> 65 and funding \> 0.03%\",
>
> \"source_trades\": \[\"trade_142\", \"trade_156\"\],
>
> \"score\": 3,
>
> \"active\": true
>
> }
>
> \]
>

**10.2.3 Self-Correcting Counters**

Every rule has a score counter. Each time the rule would have correctly
prevented a loss: increment (+1). Each time the rule would have
prevented a winner: decrement (-1). When the score goes below -2, the
rule is deactivated. This prevents the system from accumulating stale
lessons that block valid setups after a regime change.

**10.3 Loop 3: Cross-Bot Intelligence (Per-Symbol, Real-Time)**

Multiple bots may run on different timeframes for the same asset. A
shared cross_bot_signals table allows bots to see each other's recent
decisions.

**Example:** A 1h BTC bot queries the table and sees: \"4h BTC bot went
SHORT 2 cycles ago with 0.85 conviction.\" ConvictionAgent factors this
in --- going LONG against a high-conviction 4h SHORT is a strong
negative signal.

Implementation: each bot writes its last decision + conviction score to
the table after every cycle. ConvictionAgent queries it for the same
symbol on different timeframes. No extra LLM calls. For B2B API, this
becomes a premium feature: multi-timeframe signal coherence.

**10.4 Loop 4: Regime History (Per-Bot, Medium-Term)**

Tracks regime classifications over time as a ring buffer of the last 20
regime labels per bot. Injected into ConvictionAgent as a sequence.

**Why this matters:** A market that has been RANGING for 20 cycles and
just detected a breakout pattern is far more significant than one that
has been RANGING for 2 cycles. Regime transition points are the most
actionable moments. ConvictionAgent can detect: \"Regime has been
RANGING for 15 cycles. TrendAgent now shows breakout forming. This is a
potential regime transition --- increase conviction if pattern
confirms.\"

The v1.x system uses a static forecast of \"next 3 candles\" for all
market conditions. This is suboptimal because the appropriate prediction
horizon depends on market regime. A ranging market should predict
mean-reversion (short horizon), while a trending market should predict
continuation (longer horizon).

                                 range extreme?              quick; short horizon

trying to predict.

**11.2 Horizon Override by Sentinel**

When the Sentinel triggers a full analysis (proactive mode), it can
override the default horizon based on the trigger condition. If the
Sentinel fired because of a volume spike at a key level, the horizon
might be shortened to 1--2 candles regardless of regime, because the
immediate reaction matters more than the trend projection. The override
is passed through to the signal agents as context.

**PART III --- INTELLIGENCE**

**13. Offline Intelligence Agents (External MCP)**

The live trading loop must never be blocked by slow computation.
However, the system benefits enormously from deep statistical analysis
and macro awareness that requires minutes or hours of processing. The
solution: external MCP (Model Context Protocol) agents that run offline
on separate infrastructure, produce static output files, and hand off
results for the live engine to consume at zero runtime cost.

Both agents run as sandboxed MCP sessions with Python code execution
capabilities. They can read from the PostgreSQL database, fetch external
data via APIs, write and execute analysis scripts, and produce
structured JSON outputs. They have no access to the live trading engine
or exchange APIs --- they are read-only analysts that publish findings.

**13.1 Overnight Quant (Alpha Mining Agent)**

Runs nightly as a cron job on a dedicated worker (e.g., Hetzner CX11,
\~€3/mo). Its purpose is to discover statistically significant trading
patterns from historical data and produce an alpha factors file that
ConvictionAgent reads at cycle start.

**13.1.2 Safety Constraints**

The Overnight Quant can only write to designated output files
(alpha_factors.json). It cannot modify code, configuration, or the
database. All outputs include sample size (n) and statistical
significance --- ConvictionAgent ignores factors with n \< 15 or low
confidence. Factors expire after 30 days unless refreshed by a
subsequent nightly run. The agent logs its full reasoning and code to
LangSmith for audit.

across \> 50% of watched assets simultaneously. In these scenarios, the
MCP agent runs an emergency assessment and publishes updated parameters
within minutes.

Data is the permanent competitive advantage. Every competitor can copy

**14.1 Layer 0: Raw Market Data (The Environment)**

What the market looked like at the moment of each decision. OHLCV
candles (full array, not just latest), volume profile, order book
snapshot (if available), funding rate, OI, liquidation levels, parent
timeframe candles, and macro regime state. This is the environment state
for RL training --- what the agent observed before acting.

**Storage:** Compressed Parquet files organized by
symbol/timeframe/date. \~5KB per cycle.

**14.2 Layer 1: Sensory Inputs (What the System Saw)**

The exact inputs each agent received, reproducible down to the byte. All
computed indicator values (RSI, MACD, Stoch, WillR, ROC, ATR, ADX, BB),
flow data snapshot, external signal values, swing levels detected,
parent TF trend computation. For vision agents: the actual chart images
(PNG) sent to PatternAgent and TrendAgent. These images are invaluable
for training a vision model later --- each image is labeled with the
agent's assessment and the eventual trade outcome.

**Storage:** Images in object storage (S3-compatible or MinIO). \~200KB
per image, 2 images per cycle. Numerical data in PostgreSQL JSONB.

**14.3 Layer 2: Cognitive Process (How the System Thought)**

The full reasoning chain that produced each decision. Complete prompts
sent to each agent (system prompt version + user message), raw LLM
responses (full text, not just parsed signal), parsed structured
signals, ConvictionAgent full reasoning trace, DecisionAgent full
reasoning trace, which learned rules fired and their scores, cross-bot
signals received, and regime history at time of decision.

This is the richest layer. It captures not just what the system decided
but why, and what alternative reasoning it considered. For supervised
fine-tuning (SFT), this provides input→output pairs. For distillation,
the teacher model's full reasoning becomes the training signal for a
smaller student model.

**Storage:** PostgreSQL cycles table with JSONB columns. Raw LLM
responses stored in full. \~30KB per cycle.

**14.4 Layer 3: Action Data (What the System Did)**

The precise actions taken. Trade direction, size, entry price, SL/TP
levels placed, order IDs and exchange responses, fill prices vs.
expected prices (slippage measurement), subsequent adjustments (SL
tightening, pyramiding, CLOSE_ALL), and timestamp of every action to
millisecond precision.

**Storage:** PostgreSQL trades table with complete order history. \~5KB
per trade.

**18. Caching Architecture**

The system has multiple components that re-compute identical or
near-identical data. A multi-layered cache reduces data fetches by
\~10x, cuts external API costs proportionally, and eliminates redundant
computation. LLM responses are explicitly NOT cached --- market context
changes even when inputs look similar, and caching reasoning creates
dangerous false confidence.

**18.1 Cache Layers**

  --------------------------------------------------------------------------------------------
  **Layer**   **What Is Cached**  **Key Pattern**             **TTL**         **Savings**
  ----------- ------------------- --------------------------- --------------- ----------------
  L1: Symbol  OHLCV candles,      symbol:timeframe            One candle      10 bots on same
  Data        computed                                        period          symbol = 1 fetch
              indicators, swing                               (15m--1d)       instead of 10
              levels                                                          

  L2: Flow    Funding rate, OI,   flow:symbol                 30 seconds      All bots on same
  Data        OI delta,                                                       symbol share one
              liquidations                                                    flow snapshot

  L3:         QuantHQ regime,     provider:symbol             1--4 hours      Rate limit
  External    sentiment scores,                                               protection +
  APIs        news                                                            cost saving

  L4: LLM     System prompt       Prompt hash                 Session-based   \~90% cost
  Prompts     tokens (Anthropic                                               reduction on
              built-in)                                                       cached tokens

  L5: Chart   Base candlestick    chart:symbol:tf:timestamp   One candle      Pattern + Trend
  Images      rendering + axis                                period          agents share
              params                                                          base render

  L6:         Rolling indicators, In-memory per Sentinel      Continuous      O(1) updates
  Sentinel    incremental                                     (incremental)   instead of O(n)
  State       computation                                                     recompute

+-----------------------------------------------------------------------+
| **Why LLM Responses Are NOT Cached**                                  |
|                                                                       |
| Even when indicator values look identical across two cycles, the      |
| market context has changed: time has passed, candles have formed,     |
| flow data has shifted. Caching an LLM's directional assessment from   |
| 30 minutes ago and reusing it now would give false confidence in      |
| stale reasoning. Every analysis cycle deserves fresh LLM evaluation.  |
| The cost savings from caching data inputs (\~\$0.015/cycle in reduced |
| prompt tokens + eliminated redundant API calls) are substantial       |
| without needing to cache the reasoning itself.                        |
+-----------------------------------------------------------------------+

**18.2 Implementation**

Single-server deployment: Python cachetools with TTL for in-process
caching, shared dictionary for cross-bot caching within the same
process. Multi-server deployment: Redis. All cache keys go to Redis,
allowing any worker to read cached data written by another worker. Redis
also serves as the message queue for Sentinel events, so it is already
in the infrastructure stack.

**18.3 Cache Manager Interface**

> class CacheManager:
>
> def \_\_init\_\_(self, backend: str = \'memory\'):
>
> self.store = RedisCache(url) if backend == \'redis\'
>
> else MemoryCache()
>
> self.metrics = CacheMetrics() // hits, misses, savings
>
> async def get_or_fetch(self, key: str, ttl: int,
>
> fetch_fn: Callable) -\> Any:
>
> cached = await self.store.get(key)
>
> if cached:
>
> self.metrics.record_hit(key)
>
> return cached
>
> self.metrics.record_miss(key)
>
> result = await fetch_fn()
>
> await self.store.set(key, result, ttl=ttl)
>
> return result

Cache metrics (hit rate, miss rate, estimated cost savings) are exposed
in the System Health dashboard and tracked in the tracking system for
optimization.

**18.4 Cache Invalidation Rules**

Caches invalidate on three conditions: TTL expiry (automatic), new
candle close (triggers recomputation of indicators, charts, swing
levels), and manual flush (operator can clear all caches via dashboard
for debugging). The Sentinel maintains its own in-memory state and never
reads from shared cache --- it needs the freshest possible data for
real-time readiness scoring.

**19.4 LangSmith Integration**

All LLM calls across the platform are traced via LangSmith with
per-agent, per-bot, per-user trace separation. In the v2.0 architecture,
ConvictionAgent and ReflectionAgent traces are particularly important
for debugging conviction calibration and rule quality. For B2B
customers, LangSmith-style trace access is offered as a premium
transparency feature: customers can see the full reasoning chain behind
every signal they receive.

Each trace includes: agent name, prompt version, full input (system +
user prompt, truncated for readability), full output (raw LLM response),
input/output token counts, estimated cost in USD, latency in
milliseconds, and cache hit status. Traces are organized by: run (single
LLM call), cycle (all 5 agent calls in one analysis), and bot (all
cycles for a TraderBot's lifecycle).

LangSmith enables: debugging agent behavior on specific inputs, comparing
prompt versions (A/B testing), tracking cost per agent over time,
identifying slow or expensive calls, and reproducing any historical
decision for the data moat.

Configuration: ``LANGCHAIN_TRACING_V2=true``, ``LANGCHAIN_API_KEY``, and
``LANGCHAIN_PROJECT=quantagent-v2`` in ``.env``. Tracing is conditional
--- if the environment variable is not set or langsmith is not installed,
all LLM calls proceed normally without tracing overhead.

**20. Security & Authentication**

Financial platform handling exchange API keys, user accounts, and real
money. Security is a first-class architectural concern, not an
afterthought.

**20.1 User Authentication**

JWT-based authentication with refresh tokens for web and mobile. OAuth2
social login (Google, Apple) for consumer users. API key authentication
for B2B customers with per-key scoping: read-only, signals-only, or full
access. All sessions have configurable expiry and can be revoked.

**20.2 Exchange Credential Security**

**At rest:** Exchange API keys encrypted with AES-256 in PostgreSQL.
Encryption key stored in environment variables or a secrets manager
(HashiCorp Vault for production), never in code or database.

**In transit:** All API communication over TLS 1.3. No plaintext
transmission of credentials at any point in the pipeline.

**On mobile:** iOS Keychain for local credential storage. Keys
transmitted to backend only during initial setup over TLS. Biometric
authentication (Face ID / Touch ID) required to access the app.

**Access control:** Exchange credentials accessible only by the user's
own Sentinel and TraderBots. No admin access to plaintext keys.
Credentials are decrypted in memory only for the duration of an API call
and never logged.

**20.3 Multi-Tenant Data Isolation**

User A's data must never leak to User B. This is enforced at every layer
of the system:

**Database:** Every table includes user_id. All queries filter by
authenticated user. Row-level security policies in PostgreSQL enforce
this at the database level, not just application level. Cross-bot
signals filter by user_id --- only same-user bots share intelligence.

**Object storage:** Chart images and data moat files stored in
user-scoped directories (user_id/symbol/date/). No shared directories
between users.

**WebSocket:** Dashboard streams are user-scoped. Sentinel events only
pushed to the owning user's active connections. Connection
authentication verified on every message, not just on connect.

**B2B API:** API keys are customer-scoped. Each key has its own rate
limits and usage metering. Response data never includes other customers'
information.

**Logging:** All logs are scrubbed of sensitive data (API keys, wallet
addresses, user PII) before storage. Structured logging with user_id for
audit but no credential exposure.

**20.4 GDPR Compliance**

The founder is based in the Netherlands --- GDPR applies to all EU users
and any platform serving EU residents. Required implementations:

**Data export:** User can request a full export of all their data
(trades, rules, cycle history, chat logs) in machine-readable format
(JSON). Must be fulfilled within 30 days.

**Data deletion:** Complete removal of all user data including trades,
reflection rules, training examples derived from their activity, and
chat history. The data moat must support selective deletion without
corrupting other users' data.

**Privacy policy:** Clear disclosure of what data is collected, how it
is used (including for ML training), who has access, and data retention
periods.

**Cookie consent:** Landing page and dashboard must implement cookie
consent banners for EU visitors.

**Data processing agreements:** Required for every third-party API that
receives user data (Claude API, exchange APIs, external signal
providers).

Paper trading uses the Hyperliquid testnet adapter. The engine runs the

testnet with simulated balances. This is not a mock --- it is a real
exchange environment with real order matching, just fake money. All
adapter methods (orders, SL/TP, positions, funding rates) work
identically on testnet. The paper trading experience is visually
identical to live trading --- the user cannot tell the difference until
they check their real exchange balance.

flag. This data is useful for backtesting validation (comparing paper
results to what would have happened live) but is excluded from ML
training to prevent contamination with simulated fills.

positive, the upgrade prompt is confident: "Your AI trader made \$X last
week. Ready to go live?" If paper P&L is negative, the prompt is honest:
"The market was tough this week. Your AI limited losses to \$X while the
market dropped Y%. Want to continue paper trading or go live with
conservative settings?"

**28. Software Versioning Strategy**

**28.1 Engine Core (CalVer + SemVer Hybrid)**

The engine uses a hybrid versioning scheme that combines Calendar
Versioning (CalVer) for time-based context with Semantic Versioning
(SemVer) for compatibility guarantees. Format:
YYYY.MM.MAJOR.MINOR.PATCH.

The CalVer prefix (YYYY.MM) communicates freshness --- critical for a
signal product where the intelligence improves continuously via
reflection rules, alpha factors, and prompt refinements. The SemVer
suffix (MAJOR.MINOR.PATCH) communicates compatibility --- critical for
B2B clients who need stability guarantees. Example: 2026.04.2.1.3 means
April 2026, second major architecture, first feature addition, third
bugfix.

  --------------------------------------------------------------------------
  **Component**   **When It Changes**                   **Example**
  --------------- ------------------------------------- --------------------
  YYYY.MM         Automatically each calendar month     2026.04 → 2026.05

  MAJOR           Breaking changes to engine API        2026.04.1.x.x →
                  contract or fundamental architecture  2026.04.2.0.0
                  redesign                              

  MINOR           New features, agents, signal          2026.04.2.0.x →
                  producers, exchange adapters          2026.04.2.1.0

  PATCH           Bug fixes, prompt improvements,       2026.04.2.1.0 →
                  parameter tuning                      2026.04.2.1.1
  --------------------------------------------------------------------------

Pre-release tags indicate stability: 2026.04.2.0.0-alpha.1 (internal
testing), 2026.04.2.0.0-beta.1 (limited external testing),
2026.04.2.0.0-rc.1 (release candidate). Build metadata
(2026.04.2.0.0+build.1234) tracks exact builds without affecting version
precedence.

**28.2 B2B API Versioning**

The API version is independent of the engine version. API URLs include
the version: /v1/analyze, /v2/analyze. When the response schema changes,
a new API version is created. Old versions remain supported for a 6--12
month deprecation period. This ensures B2B customers never break when
the engine is upgraded.

**28.3 Prompt Versioning**

Agent prompts evolve independently from code. Each prompt gets its own
version tracked in a prompt_versions table. This enables A/B testing of
prompts, rollback to previous versions without code changes, and
critically, every stored decision records which prompt version generated
it --- essential for the data moat and reproducibility.

**28.4 ML Model Versioning**

Each trained model gets a version tag: regime_model_v1.0_2026Q2. The
version is recorded alongside every signal the model produces. When a
new model is deployed, the system can compare its accuracy against the
previous version in real-time (shadow mode). Models are never deleted
--- previous versions are archived for comparison and rollback.

**PART VII --- OPERATIONS**

**29. Tracking & Observability System**

The tracking system serves four audiences: the developer/operator
(debugging and optimization), the consumer user (portfolio performance),
the B2B customer (signal quality proof), and the system itself
(self-healing and parameter tuning). All tracking data is stored in
PostgreSQL with a clean relational schema.

**29.1 Financial Tracking**

**29.1.1 Per-Trade Metrics**

Every trade records: entry price, exit price, direction (LONG/SHORT),
position size, fees paid, slippage (expected fill vs. actual fill),
realized P&L, R-multiple (profit divided by risk taken), time in trade
(seconds), number of pyramid adds, exit reason (SL hit, TP1, TP2,
trailing stop, CLOSE_ALL signal, manual close), conviction score at
entry, regime at entry.

**29.1.2 Per-Bot Metrics**

Computed in real-time from trade history: cumulative P&L (realized +
unrealized), win rate, average winner size, average loser size, profit
factor (gross wins / gross losses), largest single win, largest single
loss, max consecutive wins, max consecutive losses, Sharpe ratio
(rolling 30-day), Sortino ratio, maximum drawdown (peak-to-trough),
recovery factor (total return / max drawdown), average R-multiple,
expectancy per trade (average win × win rate -- average loss × loss
rate), total API costs incurred, net P&L after costs.

**29.1.3 Per-Portfolio Metrics**

Aggregated across all bots: total assets under management, total
realized P&L, total unrealized P&L, portfolio equity curve (time
series), daily/weekly/monthly return series, correlation matrix between
bots (are they diversified or correlated?), total fees paid across all
exchanges, total LLM API costs, total external signal API costs, net
profit after all costs.

**29.2 Decision Intelligence Tracking**

**29.2.1 Per-Cycle Capture**

Every analysis cycle records the complete decision context: all
indicator values at decision time, all agent signals with confidence
scores and reasoning, conviction score and regime classification,
ConvictionAgent full reasoning trace, flow data snapshot (funding rate,
OI, GEX if available), parent TF state, which learned rules fired, which
contradictions were detected, final action taken, and whether the
Sentinel or scheduler triggered this cycle.

**29.2.2 Signal Quality Metrics**

Tracked per agent over time: directional accuracy (was the signal
correct about the next N candles?), conviction calibration curve (do
trades at conviction 0.8 actually win more often than trades at 0.5?),
per-pattern accuracy (when PatternAgent detects an ascending triangle,
what percentage resolve bullish?), regime accuracy (when ConvictionAgent
classifies TRENDING, is the market actually trending based on subsequent
price action?), contradiction correlation (does high agent disagreement
predict losses?).

**29.2.3 Reflection Effectiveness**

Tracked for the ReflectionAgent system: total rules generated, active
vs. deactivated rules, rule accuracy (how often does a rule correctly
prevent a loss vs. incorrectly prevent a win), rules with highest
positive scores (most valuable lessons), rules that self-corrected
(score went negative and deactivated).

timestamp, affected user/bot), reported to #alerts Slack channel via the
Sprint Manager or a dedicated error reporter, recorded in the api_health
and bot_health tracking tables for trend analysis, and included in the

**30.3 Retry Strategy**

LLM API calls: exponential backoff (1s, 2s, 4s, 8s, max 3 retries).
Exchange API calls: immediate retry once, then exponential backoff (max
5 retries). Database writes: queued in memory, replayed on reconnection
(max 1000 queued writes before alerting). External signal APIs: single
retry, then mark as unavailable for this cycle and continue.

**30.4 Circuit Breakers**

If a dependency fails more than N times in M minutes, the circuit
breaker opens and stops attempting calls. This prevents cascade failures
where a down API causes timeout-induced slowdowns across the entire
system. Circuit breaker states (closed/open/half-open) are visible in
the System Health dashboard. When half-open, a single test request is
sent; if it succeeds, the circuit closes and normal operation resumes.

**31.1 Prompt Regression Testing Detail**

Prompt changes are the highest-risk modification in the system --- a
subtle wording change can flip signal direction on 20% of inputs. The
data moat makes regression testing possible: every historical cycle has
recorded inputs and outputs. When a prompt is modified:

**Step 1:** Extract 100 recent cycle inputs for the affected agent from
the data moat (indicator values, chart images, grounding context).

**Step 2:** Run each input through the new prompt version via Claude
API. Record outputs.

**Step 3:** Compare new signal directions to original signal directions.

**Step 4:** If \< 10% flip: prompt change is safe --- deploy. If 10--20%
flip: review flipped cases manually --- are the new signals actually
better? If \> 20% flip: the prompt change is too aggressive --- iterate
before deploying.

This process costs \~\$0.80--\$1.00 per regression run (100 LLM calls)
but prevents deploying a prompt that silently degrades signal quality.
The A/B testing framework (sec 6.3) provides ongoing validation after
deployment.

**31.2 Continuous Integration**

Every git push triggers: unit tests + integration tests (automated, \< 2
minutes). Every PR merge triggers: full test suite including exchange
adapter tests (\< 5 minutes). Every deployment triggers: QA Agent runs
full suite + prompt regression if any prompt files changed, reports to
#qa-reports Slack channel.

QuantAgent is built by a solo founder delegating execution to AI. The
project management framework is designed around this reality: the
founder makes decisions and sets direction (20% of time), AI handles
execution and process management (80% of work). Documentation IS the
project management --- there is no separate tool.

**33.1 Documentation-Driven Development**

Every document in the project serves a dual purpose: it is both
technical reference AND project management artifact. When a new Claude
Chat session opens, it reads the documentation and immediately has full
context. When Claude Code starts a task, it reads CLAUDE.md and
PROJECT_CONTEXT.md and knows the rules. Zero context loss between
sessions.

**34. AI Crew Architecture**

scratch, QuantAgent leverages Claude's native ecosystem (Cowork, Claude
Code, Skills, Plugins, Connectors, MCP) to create a crew of specialized
AI workers. Each "agent" is a configured Claude instance with a specific
role, custom skills, and scheduled tasks. The crew operates autonomously
while the founder focuses on direction and decisions.

**34.5 Why Claude's Ecosystem Over Custom Code**

Building a custom multi-agent project management system would take weeks
and create a second codebase to maintain. Using Claude's native
ecosystem is better for six reasons: skills are text files (not code),
easy to create and modify. Plugins bundle everything together ---
install once, works forever. Connectors (Slack, Gmail, Google Drive) are
pre-built. Sub-agents handle parallelism automatically. Scheduled tasks
run without intervention. Updates from Anthropic improve all agents
simultaneously. The founder is not building project management
infrastructure --- they are configuring it.

**34.6 GitHub Integration**

All agents with code access operate through the project's GitHub
repository. Claude Code pushes commits directly. The Documentation
Manager reads git diffs to identify what changed. The QA Engineer pulls
the latest code to run tests. The Sprint Manager reads git log to
determine task completion. The repository is the single source of truth
for code; the .md files in the repo are the single source of truth for
project state.

**35. Implementation Roadmap**

The roadmap integrates engine development, AI Crew setup, and product
launch into a single timeline. AI Crew skills are built early (Phase 1b)
so they accelerate all subsequent phases. Each phase builds on the
previous; later phases cannot start without earlier phases completing.

**Phase 1a: Core Engine Redesign (Weeks 1--2)**

Target: 2 weeks. Implement on Hyperliquid only.

1.  Implement SignalProducer interface, structured SignalOutput, and
    modular Event Bus

2.  Redesign all agent prompts: grounding context, structured output,
    contradictions, dynamic forecast horizon

3.  Build ConvictionAgent with fact/subjective labeling and regime
    classification

4.  Refactor DecisionAgent to consume ConvictionOutput instead of raw
    signals

5.  Add parent timeframe data fetching and local trend computation

6.  Build FlowAgent with CryptoFlowProvider (funding rate, OI from
    Hyperliquid)

7.  Implement dynamic risk profiles (regime-driven ATR mult, RR,
    position sizing)

8.  Add pyramid distance-to-resistance gate and mechanical safety checks

9.  Implement ML model slot interface (empty, returns null)

10. Build Sentinel system with WebSocket price feed and readiness
    scoring

**Phase 1b: AI Crew Foundation (Week 2--3)**

Target: 1 week. Build core AI Crew skills that accelerate all subsequent
development. These run from Week 3 onward, compounding output.

1.  Build quantagent-docs.skill --- auto-updates PROJECT_CONTEXT.md from
    code changes

2.  Build quantagent-qa.skill --- runs test suite, reports results to
    Slack

3.  Build quantagent-pm.skill --- reads SPRINT.md + git log, posts
    progress to Slack

4.  Build quantagent-research.skill --- weekly scan of arXiv, GitHub,
    blogs for new techniques

5.  Set up Slack workspace with channels (#sprint-updates, #qa-reports,
    #alerts, etc.)

6.  Configure Cowork scheduled tasks for QA (daily), Docs (daily),
    Sprint (daily), Research (weekly)

**Phase 2: Memory + Database + Tracking (Weeks 3--4)**

Target: 2 weeks. AI Crew (QA, Docs, Sprint Manager) now active and
assisting.

1.  Build ReflectionAgent with PostgreSQL rulebook and self-correcting
    counters

2.  Implement cross-bot intelligence table and ConvictionAgent
    integration

3.  Add regime history ring buffer

4.  Build repository pattern abstraction; implement PostgreSQL backend

5.  Integrate external signal APIs (QuantHQ, Guavy, or similar)

6.  Build comprehensive tracking system (all 5 categories) + caching
    layers

7.  Implement ephemeral TraderBot lifecycle with Orphan Position Reaper

8.  Begin data moat capture (all 6 layers recording from first live
    trade)

**Phase 3: Backtesting + MCP Agents + Signals (Weeks 5--7)**

Target: 3 weeks. Research Agent now providing weekly digests to inform
decisions.

Build backtesting framework with offline replay. Build Overnight Quant
MCP agent (nightly cron, alpha_factors.json). Build Macro Regime Manager
MCP agent (daily, macro_regime.json). Build signal distribution
(Discord + Telegram). Build quantagent-metrics.skill for automated
nightly performance analysis. Prove alpha with LLM-only engine across
multiple assets and timeframes.

**Phase 4: ML Models + Multi-Exchange + Dashboard (Weeks 8--12)**

Target: 1 month. Data moat now has 5+ weeks of labeled data.

Train RegimeModel from collected data. Train DirectionModel (XGBoost).
Add exchange adapter capabilities system. Implement additional exchange
adapters (Interactive Brokers, Alpaca, Binance). Build B2B API layer
with auth, rate limiting, metering. Begin SolidJS dashboard rebuild.
Build quantagent-seo.skill, quantagent-marketing.skill, and
quantagent-blog.skill. Launch landing page with live performance
metrics. Activate daily blog content pipeline (Research Agent → Blog
Writer → SEO Agent → publish). Target: 30 blog posts by end of Phase 4.

**Phase 5: Product Launch + Scale (Weeks 12--24)**

Target: 3 months. Full AI Crew operational, all skills active.

Complete SolidJS dashboard migration. Build iOS app (TestFlight then App
Store). Launch consumer product with managed hosting and ephemeral swarm
bot UX. Launch B2B API publicly with signal quality leaderboard. Begin
data product licensing (labeled datasets, alpha factor feeds). Train
AnomalyDetector. Begin distillation of Claude decisions into fast
student models. RL-based parameter optimization.
