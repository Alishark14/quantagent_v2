- **Sentinel escalating readiness threshold + SetupResult feedback loop (Task 11)** — Implements the post-SKIP threshold ratchet from the user's spec, tying it back to BotManager via a new `SetupResult` event so the Sentinel finally gets feedback about what the analysis pipeline did with its triggers. Without this loop, every readiness spike that the pipeline rejected as SKIP would re-fire on the next tick at the same threshold, wasting LLM budget. **`engine/events.py`**: new `SetupResult(Event)` carrying `symbol`, `outcome` ("TRADE" / "SKIP"), `action` (literal `TradeAction.action`), `bot_id`, `conviction_score`. Outcome is the canonical TRADE/SKIP classification used by Sentinel escalation. Docstring spells out the rules so future readers don't have to rediscover them: TRADE = `action ∈ {LONG, SHORT, ADD_LONG, ADD_SHORT}` AND `order_result.success == True`; SKIP = anything else (SKIP / HOLD / CLOSE_ALL / errored bot / failed order / missing order_result). **`sentinel/config.py`**: 4 new module-level constants pinning the spec defaults — `BASE_READINESS_THRESHOLD = 0.30`, `ESCALATION_STEP = 0.10`, `MAX_ESCALATION = 0.25` (ceiling = 0.30 + 0.25 = 0.55), `SKIP_COOLDOWN_SECONDS = 900` (15 minutes). New `get_sentinel_escalation_config()` accessor returns a defensive copy of all four as a dict so tests + callers can grab the bag without importing each constant individually. **`sentinel/monitor.py`** (SentinelMonitor refactor): four new constructor params with config-driven defaults — `base_threshold` (overrides the legacy `threshold` arg as the per-monitor base), `escalation_step`, `max_escalation`, `skip_cooldown_seconds`. Per-symbol escalation state lives in `_current_threshold: dict[str, float]` (keyed by symbol per the spec, even though each monitor is currently single-symbol — future-proofs the data layout for a multi-symbol monitor refactor). The cooldown duration that `_can_trigger` actually checks lives in `_active_cooldown_seconds`, which flips between the candle-period default (`_candle_cooldown_seconds`) and `SKIP_COOLDOWN_SECONDS` in response to SetupResult events. The legacy `_cooldown_seconds` attribute is preserved as a back-compat alias pointing at the candle-period default so the existing 5 `TestSentinelMonitorDynamicDefaults` tests don't have to be touched. New methods: `current_threshold(symbol=None)` (public read of the escalated value with base fallback), `subscribe_results()` (registers `_on_setup_result` on the bus — call once during wiring), `_on_setup_result(event)` (filters by symbol, dispatches to escalate/reset helpers, defensively logs+ignores unknown outcomes), `_escalate_threshold_after_skip` (raises threshold by `ESCALATION_STEP`, capped at `BASE + MAX_ESCALATION`, switches to SKIP cooldown), `_reset_threshold_after_trade` (resets to base, restores candle-period cooldown), `_reset_all_escalations_for_new_candle(new_ts)` (called from `_check_once` when the latest candle's timestamp advances). Candle-close detection lives in `_check_once` via `_last_candle_ts`: the first observation initializes the marker WITHOUT resetting (a fresh-start Sentinel shouldn't immediately wipe escalation state it never built), and subsequent observations only reset when `latest_candle_ts > _last_candle_ts`. The trigger gate at the top of `_check_once` reads `current_threshold()` (the escalated value, falling through to base) instead of the legacy `self._threshold` — this is the only place the threshold gate is evaluated, so escalation flows through one site cleanly. Public read-only properties: `base_threshold`, `active_cooldown_seconds`. **`engine/bot_manager.py`** (BotManager closes the loop): new module-level `_TRADE_ACTIONS` frozenset = `{LONG, SHORT, ADD_LONG, ADD_SHORT}` (the canonical TRADE action set). New static `_classify_outcome(result)` maps a TraderBot result dict to "TRADE" / "SKIP" via the rules in the SetupResult docstring. New async `_publish_setup_result(symbol, bot_id, result)` builds + publishes the event, with try/except so a publication failure logs but never propagates (the bot already finished, the cleanup is done, Sentinel feedback is best-effort). Both `_run_bot` (event-triggered path) AND `spawn_bot` (manual/scheduled path) call `_publish_setup_result` AFTER the active-set lock is released — the handler chain may call into Sentinel state, and we don't want it sitting on our mutex. The concurrency-blocked path in `spawn_bot` does NOT publish: the pipeline never ran, so there's no SKIP/TRADE outcome to feed back, and emitting SKIP would incorrectly escalate the Sentinel for an orthogonal reason. **Spec deviations to flag**: (1) The user's spec said `self._base_threshold: float = 0.30  # from config, existing value` — but the legacy `SentinelMonitor.threshold` constructor default was `0.7`, NOT 0.30. I preserved the legacy 0.7 as the back-compat default when callers don't pass `base_threshold` (so existing tests + scripts using the old `threshold=` arg keep working), and added `base_threshold` as a separate first-class arg with a `None` default that falls through to the legacy. The SetupResult approach is exactly one publish per spawn, with no timing assumptions. (3) `CLOSE_ALL` is classified SKIP for escalation purposes — a CLOSE_ALL closes a position rather than opening one, so there's no new-entry rationale to reset Sentinel's escalation. - **FlowSignalAgent + N-signal-aware ConvictionAgent (Task 9)** — Elevates flow from a passive data dump in the grounding header into a 4th signal voice that participates in the conviction prompt alongside Indicator/Pattern/Trend. **`engine/data/flow/signal_agent.py`** (new): `FlowSignalAgent(SignalProducer)` — code-only, zero LLM cost, ~microsecond latency.
