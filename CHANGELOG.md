Inside `_check_once`, after the readiness score is computed AND the cooldown / daily-budget gate has approved the trigger, `_active_blackout_reason()` is called: it loads the macro file via `load_macro_regime`, honours the `expires` timestamp (a stale regime overlay should NOT keep blocking new entries indefinitely), walks `blackout_windows`, and returns the reason if any contain `now`. On hit: a single `INFO`-level log line (`"Sentinel: {symbol} setup detected (readiness={score:.2f}) but blackout active ({reason}), suppressing"`) and `return` ŌĆö NO event is emitted, NO daily budget counter is incremented, NO `_last_trigger` is stamped. This is a deliberate deviation from a literal "before emitting" reading of the spec: a blackout that consumes 8 daily triggers would prevent any post-blackout setups from firing, which is exactly the wrong behaviour.
